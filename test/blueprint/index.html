2. ユーザーリサーチとペルソナ設計（ユーザーを深く知る）

🕵️ このステップで何をするの？

簡単に言うと：「実際にサイトを使う人たちのことをめちゃくちゃ深く知る」です

例えば、友達にプレゼントを選ぶ時、「その人が何を好きか」「どんな悩みを持っているか」を知ってから選びますよね？サイト作りでも同じで、「実際に使う人」のことを深く理解してから作ります。

📝 「ペルソナ」って何？

ペルソナとは、「代表的なユーザーの人物像」のことです。たとえば「田中さん（50歳）はこんな悩みを持っていて、こんなことを求めている」という具体的な人物像を作ります。こうすることで、「田中さんが使いやすいサイト」を作ることができます。

3. ユーザージャーニーマップの作成（使い方の流れを作る）

💡 専門用語の解説

  • セルフ型マッチング: 人間が仲介せず、ユーザー同士が直接やり取りできる仕組み
  • 逆スカウト: 普通は企業→CFOだけど、CFO→企業からもアプローチできること
  • 両面マーケティング: 企業とCFO、両方に同時に宣伝すること
  • マネタイズ: お金を稼ぐ仕組み
  • EXIT: 会社を売却すること（M&AやIPO）
  • CxO: CEO、CFO、CTOなど、会社の役員クラスの人たち

KPI測定ポイント

  • いいね率: いいねアクション数 ÷ 表示数
  • スカウト率: スカウト送信数 ÷ いいね数
  • 面談率: 面談設定数 ÷ スカウト数
  • 契約率: 契約締結数 ÷ 面談数

💡 わかりやすく言うと...

KPI = サービスがうまくいってるかを測る数字

  • いいね率: 100人がCFOを見て、何人が「この人いいな」と思うか
  • スカウト率: 「いいな」と思った人の中で、何人が実際にメッセージを送るか
  • 面談率: メッセージを送った人の中で、何人が「会いましょう」となるか
  • 契約率: 実際に会った人の中で、何人が「一緒に働きましょう」となるか

→ つまり、どの段階で人が諦めちゃうかがわかる仕組み！

💡 Firebase認証 + Supabaseデータベースの理由

なぜこの組み合わせがベスト？

  • Firebase Authentication:
    • Google OAuth認証が簡単に実装できる
    • メール/パスワード認証も安定
    • セキュリティが強固（Googleが管理）
    • ユーザー管理画面が使いやすい
  • Supabase Database:
    • PostgreSQLで複雑なクエリが書きやすい
    • JSONカラムでスキル情報を柔軟に保存
    • Row Level Security（RLS）で安全
    • リアルタイム機能でメッセージ対応

→ 認証はFirebaseの得意分野、データ管理はSupabaseの得意分野を使い分け！

🔄 認証フロー（Firebase → Supabase連携）
  1. ユーザーがFirebaseでログイン（Google OAuth or メール）
  2. FirebaseがIDトークンを発行
  3. Next.js APIでIDトークンを検証
  4. 検証後、Supabaseにユーザーデータをupsert
  5. 以降はSupabaseのデータを使ってサービス提供

💰 MVP無料構成案

前提：NEXT.js、Vercel、SendGrid、Supabase のいずれも無料プランを組み合わせて、RIGHTARM の初期フェーズを立ち上げることは十分に可能です。MVP（Minimum Viable Product）段階では、開発コストを抑えつつ早期にプロトタイプを市場に投入し、ユーザー反応を収集するのが狙いです。

無料プランでの構成案

  • Next.js on Vercel
    • 概要：React ベースのフレームワーク Next.js を、Vercel の無料ホスティングプランにデプロイ
    • メリット：自動デプロイ、無料 SSL、サーバーレス関数も無料枠で利用可能
  • Supabase（無料プラン）
    • 概要：PostgreSQL データベース、Auth（認証）、Storage、Edge Functions を無償で提供
    • メリット：データベース容量・認証ユーザー数の無料枠で MVP をカバー
  • SendGrid（無料プラン）
    • 概要：12,000 通／月 のトランザクションメールを無料送信可能
    • 用途：サインアップ確認メール、パスワードリセット、通知メールなどの SMTP サーバーとして利用
  • Vercel サーバーレス関数 or Supabase Edge Functions
    • 概要：NEXT.js 画面上の API を Vercel Functions で処理しつつ、認証トークンの発行やカスタム通知は Supabase Edge Functions に振り分け

📊 各サービスの無料枠と留意点

  • Next.js / Vercel
    • ビルド数や帯域幅に上限あり（無料プランで試作から小規模運用までは十分）
    • 高トラフィック時はパフォーマンス低下の可能性
  • Supabase
    • データベースサイズ：500MB 無料枠
    • Auth ユーザー数：50,000 MAU（Monthly Active Users）無料枠
    • Edge Functions：125,000 実行/月 まで無料
  • SendGrid
    • 月間 12,000 通のメール送信が無料
    • 大量ユーザー獲得でメール数が増えると超過リスク
  • 注意事項
    • いずれも無料枠を超過すると「メール未送信」「API レスポンス低下」「デプロイ不可」などの障害リスク
    • リソース使用量は定期的にモニタリングし、必要に応じて有料プランへのアップグレードを検討

⚙️ 設定と運用フロー

  1. 各アカウント作成
    • Vercel、Supabase、SendGrid で無料アカウント登録
    • SendGrid で API キー発行
  2. Supabase に SMTP 設定
    • Supabase コンソール → Authentication → Settings → SMTP に SendGrid 情報を登録
  3. Next.js プロジェクト設定
    • `@supabase/supabase-js` をインストールし、環境変数に Supabase URL / ANON_KEY を設定
    • Vercel のプロジェクト環境変数にも同じく登録
  4. 認証フロー実装
    • Supabase Auth でマジックリンク or メール+パスワードログインを有効化
    • Next.js の API Routes 経由で `supabase.auth.signInWithOtp({ email })` を呼び出し
  5. デプロイ＆テスト
    • Vercel にプッシュ → 自動デプロイ
    • 本番ドメイン・SSL 構成済みの状態で動作確認

まとめ：無料プランだけで MVP を構築し、ユーザーからのフィードバックを得ることは十分に可能です。ただし、トラフィック増大やユーザー数急増に伴い無料枠がつきるリスクもあるため、定期的に利用状況をモニタリングし、本格展開前に有料プラン移行のタイミングを検討してください。

  • 技術: Stripe Connect（Standard or Express）使用
  • フロー: 企業→プラットフォーム→CFO（自動振込）

3. 成果報酬型（融資・補助金・EXITなど）

  • 融資: 融資額の3%
  • 補助金: 補助金額の5%
  • フルタイム採用転籍: 年収の20%
  • EXIT: 相場5-10%
  • 報告: CFO側に成果報告義務、契約上企業にも報告義務を明記

💡 収益モデルをわかりやすく説明

要するに、RightArmが儲かる3つのパターン：

  1. 最初に手数料をもらう方式: 企業がCFOと契約したら、契約金額の5%を最初にいただく
  2. 毎月少しずつもらう方式: 毎月の支払いの3%をプラットフォーム利用料としていただく
  3. 成功したらもらう方式: CFOが融資や補助金を取ってきたら、その金額の一部をいただく

→ つまり、マッチングが成功して、実際に成果が出た時だけお金をいただく仕組み！

9. 認証システム設計

RightArm プラットフォームの認証システムは、Firebase Authentication と Supabase データベースを連携させた安全で拡張性の高いシステムを構築します。

重要: この認証システムは、youbou.md要件のUI/UX実装が完了後の二次実装項目です。まずはセクション10のUI/UX要件を最優先で実装してください。

🔐 認証システム概要

Firebase でユーザー認証を行い、Supabase にユーザー情報を安全に保存する全体の流れ：

  1. ユーザーがメールや Google アカウントでログイン
  2. Firebase が「IDトークン」という鍵を発行
  3. その鍵をサーバーに渡し、サーバー側で正当性を確認
  4. 確認後、Supabase にユーザー情報を登録

💡 Google OAuth認証の魅力

なぜGoogle OAuth認証を追加するの？

  • ユーザビリティ: 「Googleでログイン」ボタン1つで簡単登録
  • 信頼性: パスワードを覚える必要がない
  • セキュリティ: Googleが二段階認証も管理
  • プロフィール情報: 名前、メール、プロフィール画像を自動取得
  • ビジネス向け: CFO・経営者はGoogleアカウントを必ず持っている

→ 特にビジネス系サービスでは、Google OAuthは登録率が2-3倍向上！

📈 KPI自動計算の仕組み

🔧 実際のSQL例（計算方法）

いいね率を計算する場合
-- 今月のいいね率を計算
SELECT 
  COUNT(CASE WHEN action_type = 'view' THEN 1 END) as 表示数,
  COUNT(CASE WHEN action_type = 'like' THEN 1 END) as いいね数,
  ROUND(
    COUNT(CASE WHEN action_type = 'like' THEN 1 END) * 100.0 /
    COUNT(CASE WHEN action_type = 'view' THEN 1 END), 
    2
  ) as いいね率
FROM user_actions 
WHERE created_at >= '2024-01-01' 
  AND created_at < '2024-02-01';
ユーザー別の成約率を見る場合
-- CFO別の成約率ランキング
SELECT 
  cp.display_name as CFO名,
  COUNT(CASE WHEN ua.action_type = 'view' THEN 1 END) as 表示回数,
  COUNT(CASE WHEN ua.action_type = 'like' THEN 1 END) as いいね数,
  COUNT(CASE WHEN ua.action_type = 'contract' THEN 1 END) as 契約数,
  ROUND(
    COUNT(CASE WHEN ua.action_type = 'contract' THEN 1 END) * 100.0 /
    COUNT(CASE WHEN ua.action_type = 'view' THEN 1 END), 
    2
  ) as 総合成約率
FROM user_actions ua
JOIN cfo_profiles cp ON ua.target_user_id = cp.user_id
GROUP BY cp.user_id, cp.display_name
ORDER BY 総合成約率 DESC;

    • BIツール: データを分析・可視化するソフト
    • ERP: 会社全体の業務を管理するシステム
    • リスケ: 借金の返済条件を変更してもらうこと
    • 内部統制: 会社が不正や間違いを防ぐための仕組み
