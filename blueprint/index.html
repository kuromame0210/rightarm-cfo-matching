1. 目的と範囲の定義

まずは何のために使用設計を行うのか、どこまでを手順書の対象とするのかを明確にします。具体的には以下の観点で整理します。

✅ 確定事項

  • サービス名: RIGHTARM
  • ミッション: 全国の企業に、"右腕CFO"を届ける
  • プラットフォーム導入目的: CFO⇔企業間マッチングの最適化
  • 対象ユーザー: 企業担当者、CFO、管理者
  • 収益モデル: 契約金額の5%仲介手数料

成果物のイメージ

  • ユーザーフロー図: 登録→検索→スカウト→面談→契約→支払いの流れ
  • ワイヤーフレーム: ホーム、検索結果、プロフィール詳細、メッセージ画面、管理画面
  • プロトタイプ: 主要機能の操作感確認用

3. ユーザージャーニーマップの作成

ペルソナごとにサービス利用の一連の流れを可視化し、各タッチポイントでのユーザーの行動・感情・要件を整理します。

✅ 確定ユーザーフロー

  • 企業側: 登録（10項目程度）→ CFO検索・フィルタリング → いいね → スカウト送信 → 面談設定 → 契約 → 支払い（手動振込）
  • CFO側: 登録（10項目程度）→ 企業検索・フィルタリング → プロフィール確認 → スカウト受信 → 面談設定 → 契約 → 支払い受領
  • 管理者側: ユーザー管理 → 支払い確認 → ステータス更新 → KPIモニタリング

KPI測定ポイント

  • いいね率: いいねアクション数 ÷ 表示数
  • スカウト率: スカウト送信数 ÷ いいね数
  • 面談率: 面談設定数 ÷ スカウト数
  • 契約率: 契約締結数 ÷ 面談数

実施内容

  • 上記フローを詳細化し、各フェーズでのユーザーの感情変化をマッピング
  • 各フェーズで発生しうる障壁（入力項目の多さ、通知漏れなど）と改善案を併記
  • 優先度の高い改善ポイントを抽出し、バックログとして管理

4. ワイヤーフレーム設計

画面ごとの構成要素とレイアウトを決定し、実際の画面設計に向けた青写真を作成します。

✅ 確定タグ・プロフィール設計

  • 資金調達フェーズ: シード、シリーズA、シリーズB以降
  • 専門領域: M&A支援、内部統制構築、IR、タックスプランニング
  • 業界経験: IT／テック、バイオ、製造、小売り
  • 稼働条件: 週1日～5日、リモート可否、短期／長期契約

🏷️ タグ・プロフィール充実のこだわりポイント

  • マッチング精度の向上: タグ付けはマッチング精度の肝として、初期はシンプル運用後に階層化・拡張
  • 検索・フィルター最適化: ユーザーが直感的に使えるタグ体系の構築
  • 将来の拡張性: 管理画面からタグの追加・編集が可能な柔軟な設計

🎨 UX/UI設計のこだわりポイント

  • 登録フロー最適化: 10項目程度に限定し、離脱率を最小化
  • 直感的な検索UI: フィルターUIを直感的に配置、未読／既読ステータス明確表示
  • 利便性向上: カレンダー連携とオンライン面談リンク自動生成

実施内容

  • 主要ページ（ホーム、検索結果、プロフィール詳細、メッセージ画面、管理画面ダッシュボード）のワイヤーフレームを用意
  • キーユーザーアクション（「いいね」「スカウト」「面談予約」「支払い確認」など）を強調して配置
  • 入力項目やボタンのラベル、通知バナーの文言もプレーンテキストで定義

5. プロトタイピングとレビュー

本フェーズでは、画面デザインの磨き込みよりも「機能・操作感の検証」を優先し、デザインツールを用いずにコードベースでプロトタイプを構築します。あくまでユーザーが実際に触れるインターフェイスを通じて、エンジニアと経営陣が早期に課題を発見・改善できることを狙いとします。

実施内容

ライブコーディングによるプロトタイプ実装

  • 初期画面や主要フロー（登録→検索→メッセージ送信→面談予約）を、Next.js や React コンポーネントを用いて直接コーディングします
  • UI ライブラリや基本的なスタイルガイドに従いつつ、画面遷移とインタラクションの安定動作を最優先に組み立てます

内部レビュー（二人体制）

  • エンジニアの黒澤と社長のさく、二人でセッションを実施し、操作フロー上の死角や文言の違和感を洗い出します
  • ドキュメントやスプレッドシートでフィードバックを整理し、即座にコードに反映することで、設計と実装のズレを最小化します

実ユーザーによるユーザーテスト

  • 7月から実際の想定ユーザーにプロトタイプを触ってもらい、以下を中心に検証します：
    • 各画面で「次に何をすればよいか」が直感的にわかるか
    • 登録フォームや検索フィルターの入力・操作時のつまずきポイント
    • メッセージ送信や面談予約フローでの離脱・中断の有無
  • 定量的にはタスク完了率、各タスクに要する平均時間、重要なクリック箇所の誤操作率などをログに残し、レポートとしてまとめます

📅 スケジュール

  • 6月：本格的な開発着手（プロトタイプ作成開始）
  • 7月：ユーザーテスト開始（4～6名程度の実ユーザーに参加依頼）
  • 7月下旬～8月上旬：テスト結果の分析→改善タスクの優先順位付け

デザインの細部調整は次フェーズ（ビジュアルデザイン・高精度ワイヤーフレーム作成）にて行うため、本段階では「動く試作品」として機能要件のつながりや操作感にフォーカスします。

6. 実装仕様書の整備

プロトタイプをベースに、デザイナー・エンジニア間で共有するための詳細仕様書をまとめます。

✅ 確定技術スタック

  • データベース & 認証: Supabase（PostgreSQL + Auth）
  • フロントエンド & バックエンド: Next.js（フルスタック）
  • 通知 & メール: SendGrid または SES、Firebase Cloud Messaging
  • モバイル対応: レスポンシブデザイン + PWA（Progressive Web App）対応

💰 MVP無料構成案

前提：NEXT.js、Vercel、SendGrid、Supabase のいずれも無料プランを組み合わせて、RIGHTARM の初期フェーズを立ち上げることは十分に可能です。MVP（Minimum Viable Product）段階では、開発コストを抑えつつ早期にプロトタイプを市場に投入し、ユーザー反応を収集するのが狙いです。

無料プランでの構成案

  • Next.js on Vercel
    • 概要：React ベースのフレームワーク Next.js を、Vercel の無料ホスティングプランにデプロイ
    • メリット：自動デプロイ、無料 SSL、サーバーレス関数も無料枠で利用可能
  • Supabase（無料プラン）
    • 概要：PostgreSQL データベース、Auth（認証）、Storage、Edge Functions を無償で提供
    • メリット：データベース容量・認証ユーザー数の無料枠で MVP をカバー
  • SendGrid（無料プラン）
    • 概要：12,000 通／月 のトランザクションメールを無料送信可能
    • 用途：サインアップ確認メール、パスワードリセット、通知メールなどの SMTP サーバーとして利用
  • Vercel サーバーレス関数 or Supabase Edge Functions
    • 概要：NEXT.js 画面上の API を Vercel Functions で処理しつつ、認証トークンの発行やカスタム通知は Supabase Edge Functions に振り分け

📊 各サービスの無料枠と留意点

  • Next.js / Vercel
    • ビルド数や帯域幅に上限あり（無料プランで試作から小規模運用までは十分）
    • 高トラフィック時はパフォーマンス低下の可能性
  • Supabase
    • データベースサイズ：500MB 無料枠
    • Auth ユーザー数：50,000 MAU（Monthly Active Users）無料枠
    • Edge Functions：125,000 実行/月 まで無料
  • SendGrid
    • 月間 12,000 通のメール送信が無料
    • 大量ユーザー獲得でメール数が増えると超過リスク
  • 注意事項
    • いずれも無料枠を超過すると「メール未送信」「API レスポンス低下」「デプロイ不可」などの障害リスク
    • リソース使用量は定期的にモニタリングし、必要に応じて有料プランへのアップグレードを検討

⚙️ 設定と運用フロー

  1. 各アカウント作成
    • Vercel、Supabase、SendGrid で無料アカウント登録
    • SendGrid で API キー発行
  2. Supabase に SMTP 設定
    • Supabase コンソール → Authentication → Settings → SMTP に SendGrid 情報を登録
  3. Next.js プロジェクト設定
    • `@supabase/supabase-js` をインストールし、環境変数に Supabase URL / ANON_KEY を設定
    • Vercel のプロジェクト環境変数にも同じく登録
  4. 認証フロー実装
    • Supabase Auth でマジックリンク or メール+パスワードログインを有効化
    • Next.js の API Routes 経由で `supabase.auth.signInWithOtp({ email })` を呼び出し
  5. デプロイ＆テスト
    • Vercel にプッシュ → 自動デプロイ
    • 本番ドメイン・SSL 構成済みの状態で動作確認

まとめ：無料プランだけで MVP を構築し、ユーザーからのフィードバックを得ることは十分に可能です。ただし、トラフィック増大やユーザー数急増に伴い無料枠がつきるリスクもあるため、定期的に利用状況をモニタリングし、本格展開前に有料プラン移行のタイミングを検討してください。

📱 モバイル対応戦略

CFOマッチングサービスはビジネスユーザーが外出先でも利用する可能性が高いため、スマートフォン対応は必須です。以下の戦略でモバイルファーストのユーザー体験を提供します。

レスポンシブデザイン

  • Tailwind CSSを活用したモバイルファーストデザイン
  • ブレークポイント：スマホ（～768px）、タブレット（768px～1024px）、PC（1024px～）
  • タッチ操作に最適化されたボタンサイズ（最小44px）とタップ領域の確保
  • スワイプジェスチャーによる直感的な操作（カード切り替え、メニュー表示など）

PWA（Progressive Web App）対応

  • Next.js PWAでネイティブアプリライクな体験を提供
  • ホーム画面への追加機能（Add to Home Screen）
  • オフライン時の基本機能利用（プロフィール閲覧、下書き保存など）
  • プッシュ通知対応（新しいスカウト、メッセージ受信時）
  • Service Workerによるキャッシュ最適化

モバイル特化機能

  • カメラ連携：プロフィール写真撮影、資格証明書アップロード
  • 位置情報：地域別CFO検索、面談場所の提案
  • カレンダー連携：スマホのネイティブカレンダーアプリとの同期
  • 通話・ビデオ通話：面談予約からワンタップで通話開始
  • タッチID/FaceID：生体認証によるセキュアなログイン

パフォーマンス最適化

  • モバイル回線でも高速表示を実現する画像最適化（WebP、遅延読み込み）
  • Critical CSS inliningによる初期表示速度向上
  • API応答の最適化（必要最小限のデータ転送）
  • Service Workerによるバックグラウンド同期

主要機能詳細

  • 登録・認証機能: メール認証ベース、10項目程度、SNS連携は任意

9. 認証システム設計

RIGHTARM プラットフォームの認証システムは、Firebase Authentication と Supabase データベースを連携させた安全で拡張性の高いシステムを構築します。

🔐 認証システム概要

Firebase でユーザー認証を行い、Supabase にユーザー情報を安全に保存する全体の流れ：

  1. ユーザーがメールや Google アカウントでログイン
  2. Firebase が「IDトークン」という鍵を発行
  3. その鍵をサーバーに渡し、サーバー側で正当性を確認
  4. 確認後、Supabase にユーザー情報を登録

✅ Firebase プロジェクト設定

  • Firebase Console: 新しいプロジェクト作成（rightarm-cfo-matching）
  • Authentication 設定: メール/パスワード認証と Google 認証を有効化
  • ウェブアプリ登録: APIキーやプロジェクトIDなどの設定情報を取得
  • Firebase Admin SDK: サーバー側でのIDトークン検証用の秘密鍵を設定

✅ Supabase プロジェクト設定

  • プロジェクト作成: Supabase で新しいプロジェクトを作成（Tokyo リージョン）
  • データベーススキーマ: profiles テーブルの作成（Firebase UID を保存）
  • API キー取得: Project URL、anon key、service_role key を控える
  • RLS設定: Row Level Security でユーザーごとのデータアクセス制御

📋 データベーススキーマ設計

基本プロフィールテーブル（profiles）

  • id: Firebase UID（主キー）
  • email: メールアドレス
  • display_name: 表示名
  • photo_url: プロフィール画像URL
  • user_type: ユーザータイプ（company / cfo）
  • created_at: 作成日時
  • updated_at: 更新日時

企業プロフィール拡張テーブル（company_profiles）

  • company_name: 会社名
  • industry: 業界
  • company_size: 企業規模
  • funding_stage: 資金調達段階
  • location: 所在地
  • description: 会社概要

CFOプロフィール拡張テーブル（cfo_profiles）

  • first_name, last_name: 氏名
  • expertise: 専門領域（JSON配列）
  • experience_years: 経験年数
  • industries: 業界経験（JSON配列）
  • work_style: 稼働条件（JSON）
  • hourly_rate: 時給
  • bio: 自己紹介

🔧 フロントエンド認証実装

Firebase SDK 設定

  • Firebase 設定情報を環境変数で管理
  • メール/パスワード認証とGoogle認証の実装
  • IDトークン取得とバックエンドへの送信
  • 認証状態の監視とリダイレクト処理

⚙️ バックエンド API 設計

IDトークン検証 API（/api/auth/verify）

  • 検証成功時にSupabaseへユーザー情報を保存（upsert）
